Effect Guard & Mutex

src/App.tsx:320–334
useEffect

The queue useEffect checks analysisRunningRef — a ref-based mutex that prevents concurrent analyses. This was the fix for a deadlock where analyzingTrackId (state) in the dep array caused the effect to cancel itself on every state update.

// Ref-based guard (NOT state — avoids self-cancellation)
if (analysisRunningRef.current || analysisQueue.length === 0) return;

// Track lookup via ref (not state — avoids dep array inclusion)
const track = localTracksRef.current.find(item => item.id === nextTrackId);

analysisRunningRef.current = true; // Lock
setAnalyzingTrackId(nextTrackId);   // UI indicator
06

IPC: Read Audio File

src/App.tsx:245 → electron/preload.cjs:6 → electron/main.cjs:558
IPC Boundary

Reads the entire audio file via Electron IPC. A Map<string, ArrayBuffer> cache prevents re-reads for tracks already loaded on a deck. The main process reads with fs.readFile and returns Uint8Array, which is copied to a fresh ArrayBuffer in the renderer (Electron IPC may return SharedArrayBuffer).

// Renderer → IPC → Main Process
const bytes = await window.appAPI.readAudioFile(track.filePath);
const arrayBuffer = bytesToArrayBuffer(bytes); // copy to own ArrayBuffer
bufferCacheRef.current.set(track.filePath, arrayBuffer);

// Main process handler
ipcMain.handle('library:read-audio-file', async (_event, filePath) => {
  const file = await fs.readFile(filePath);
  return new Uint8Array(file);
});
07–08

Decode → Mixdown → Downsample

src/lib/audioAnalysis.ts:292–298
DSP

Web Audio API decodes the compressed audio (MP3/FLAC/WAV/etc.) into raw PCM. Then: stereo channels are averaged into mono, and 44.1 kHz samples are downsampled to 11,025 Hz (4× reduction) for faster analysis.

const decoded = await audioContext.decodeAudioData(arrayBuffer.slice(0));
const mono = mixdown(decoded);         // stereo → mono (average channels)
const resampled = downsample(mono, decoded.sampleRate, 11025);
09

BPM Estimation

src/lib/audioAnalysis.ts:117–172
DSP — Onset Autocorrelation

Selects a 90-second window anchored at 18% of the track (skipping intros). Computes an onset envelope from RMS energy increases (frame=1024, hop=256). Autocorrelates across lags corresponding to 70–180 BPM. Applies octave correction: doubles sub-84 BPM, halves super-170 BPM.

// Onset detection: positive RMS derivative
onsetEnvelope.push(Math.max(0, rms - previousRms));

// Autocorrelation: find dominant periodicity
for (let lag = minLag; lag <= maxLag; lag++) {
  score += smoothed[i] * smoothed[i - lag];  // dot product at lag
}

// Octave correction
if (bpm < 84) bpm *= 2;   // half-time → double
if (bpm > 170) bpm /= 2;  // double-time → halve
10

Key Estimation

src/lib/audioAnalysis.ts:220–290
DSP — Goertzel + Krumhansl-Schmuckler

Selects a 72-second window at 24%. Uses the Goertzel algorithm (single-frequency DFT) to measure power at each of 12 pitch classes across 3 octaves (MIDI 48, 60, 72). Accumulates a chroma vector. Correlates against Krumhansl-Kessler major and minor key profiles for all 12 roots (24 comparisons). Best match → key + Camelot code.

// Goertzel power at each pitch class × 3 octaves
for (let pitchClass = 0; pitchClass < 12; pitchClass++) {
  for (const octave of [48, 60, 72]) {
    chroma[pitchClass] += goertzelPower(..., midiToFrequency(octave + pitchClass));
  }
}

// Krumhansl-Schmuckler: correlate chroma with key profiles
majorScore = correlation(chroma, rotateProfile(MAJOR_PROFILE, root));
minorScore = correlation(chroma, rotateProfile(MINOR_PROFILE, root));

// Output: "Cm" → Camelot "5A"
const key = normalizeMusicalKey(`${NOTE_NAMES[bestRoot]}${bestMode === 'minor' ? 'm' : ''}`);
return { key, camelot: camelotFromKey(key), confidence };
11

Energy & Waveform

src/lib/audioAnalysis.ts:89–98, 70–87
DSP

Energy: Global RMS of all samples, scaled by 240 and clamped to 0–100. Quiet ambient ≈ 15, loud EDM ≈ 80+.
Waveform: Divides the track into 180 equal segments, records the peak amplitude in each. These 180 values drive the scrollable SVG waveform visualization.

// Energy: RMS → 0-100 scale
const rms = Math.sqrt(sum / samples.length);
return Math.round(clamp(rms * 240, 0, 100));

// Waveform: 180 peak buckets
for (let i = 0; i < 180; i++) {
  buckets[i] = peakAmplitudeInSegment(i); // 0-1
}

Effect Loop

The finally block always runs (no cancelled check — that was the original deadlock). It clears the mutex, removes the track from the queue, which triggers a re-render → the effect fires again for the next track. Known Issues

Issues Discovered While Tracing

1. Entire File Read Into Memory

fs.readFile(filePath) reads the complete audio file. A 50 MB FLAC creates ~150 MB transient pressure (Buffer + Uint8Array + ArrayBuffer copies across IPC).

2. No Analysis Cancellation

No way to abort an in-flight analysis (e.g., when scanning a new folder). The old cancelled flag was removed because it caused the deadlock. An AbortController mechanism could be added.

3. Sequential Processing

Tracks are analyzed one at a time. For 100+ tracks, parallel analysis (2–3 concurrent) would improve throughput.

4. Analysis Results Not Persisted

All data lives in React state. Closing the app loses all BPM, key, energy, and waveform data. Re-scanning requires full re-analysis.

5. Confidence Bias Constants

BPM confidence adds +0.35, key adds +0.32. These hand-tuned biases inflate scores — ambiguous tempo may report 0.6 confidence when actual certainty is low.
